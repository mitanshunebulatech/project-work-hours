This Privacy Policy applies to the WorkHours Enterprise application ("the Application"), an internal employee timesheet management system. It covers personal data collected from employees and administrators who use the Application as part of their employment.

## 2. Data Controller

## 3. What Personal Data Is Collected

| Category | Specific data | Source |
|---|---|---|
| Account information | Username, email address (optional), role | Created by an administrator |
| Authentication metadata | Hashed password, refresh token hashes, token expiry timestamps | Generated by the Application; raw passwords and raw tokens are never stored |
| Timesheet data | Project assignment, hours worked, entry date, remarks, approval status | Entered by the employee |

| Maintaining an audit trail of administrative actions | Legitimate interest (accountability, internal investigation) |

# Document 19 — GDPR Compliance Summary

*Applies if Nebula Tech IT or any of its employees/users fall within the territorial scope of the EU General Data Protection Regulation.*

## 1. Lawful Basis

As described in the Privacy Policy §4, processing is grounded in performance of the employment contract and legitimate interest. No special category data (GDPR Art. 9) is processed by the Application.

## 2. Data Subject Rights Implementation

| GDPR Right | Application support |
|---|---|
| Right of access (Art. 15) | An employee's own data is queryable via `/api/v1/profile/me` and `/api/v1/entries` (self-scoped); a formal data export covering all categories would currently require an administrator to compile data manually across the users/entries/audit tables, since no single "export my data" endpoint exists in v1 — flagged as a gap to close before relying on this for formal GDPR access requests |
| Right to rectification (Art. 16) | Administrators can correct user records via `PATCH /users/{id}`; employees can correct their own pending entries via `PATCH /entries/{id}` within the BR-03/BR-04 edit window |
| Right to erasure (Art. 17) | The Application implements soft-delete only (`deleted_at` timestamp), not hard deletion, to preserve historical reporting integrity — a genuine Art. 17 erasure request may require a manual hard-delete process outside the Application's current UI, which should be defined before this right needs to be honored in practice |
| Right to data portability (Art. 20) | Not currently implemented as a self-service feature; data would need to be extracted by an administrator via the existing CSV export endpoint (entries only, not the full data category set) |
| Right to object (Art. 21) | Handled via the organization's HR/legal process, not an in-application control |

## 3. Data Protection by Design and Default (Art. 25)

The Application's architecture supports this principle through: role-based access scoping (employees cannot query other employees' entries by design, not just by UI omission — see Document 9 §9.3/9.4), minimal data collection (no special category data, no unnecessary fields), and hashed-at-rest credentials. These are genuine architectural properties, not retrofitted controls.

## 4. Breach Notification

The Application's audit log (Document 10 §10.6) would be a primary forensic resource in the event of a suspected breach, but the Application itself has no automated breach-detection or notification mechanism — the organization must have its own incident response process to meet the 72-hour notification requirement under Art. 33, independent of anything built into this codebase.

# Document 21 — DPDP Act Compliance Checklist

*Applies given the development team's association with operations in India (per the project's Gujarat-based context); the Digital Personal Data Protection Act, 2023 governs processing of digital personal data within India.*

| DPDP Requirement | Status |
|---|---|
| Notice to Data Principal before processing (§5) | Requires the organization to provide employees with a clear, in-language notice — this Privacy Policy (Document 16) serves as the basis for that notice but should be reviewed against DPDP's specific notice-content requirements before being treated as compliant |
| Consent or legitimate use ground (§7) | Employment-context processing of timesheet data plausibly falls under DPDP's "legitimate uses" provisions for employment purposes, but this determination should be confirmed with counsel rather than assumed |
| Purpose limitation (§6) | The Application processes data only for the purposes described in Privacy Policy §4 — no secondary use (e.g. marketing, profiling) occurs in the Application as built |
| Data minimization | Satisfied architecturally — see Document 19 §3 (Data Protection by Design) reasoning, which applies equally here |
| Storage limitation (§9) | The retention schedule in Privacy Policy §5 must be finalized against DPDP's specific retention expectations, which may differ from the GDPR-oriented framing currently drafted |
| Reasonable security safeguards (§8) | Addressed by the Security Design document (Document 10) — bcrypt hashing, JWT short-lived tokens, RBAC, parameterized queries, audit logging |
| Data Principal's rights — access, correction, erasure (§11–13) | Same implementation status and gaps as documented in Document 19 §2 — no DPDP-specific self-service tooling exists beyond what was built for general access/correction support |
| Significant Data Fiduciary obligations (§10) | Likely not applicable given the Application's scale (small internal HR tool, not large-scale consumer data processing), but this classification should be confirmed against the organization's actual user volume and data sensitivity if that volume grows |
| Grievance redressal mechanism (§13) | Not yet implemented as a formal process — the organization should define an internal grievance contact/process before this checklist can be marked complete |
| Cross-border transfer restrictions (§17, as applicable) | Not applicable to the current single-region, self-hosted architecture; would need review if hosting infrastructure changes |

**Overall status:** This checklist identifies where the Application's existing architecture and documentation already support DPDP-aligned practices (security safeguards, purpose limitation, data minimization) versus where a formal compliance determination or additional process work is still required (notice content review, grievance mechanism, Significant Data Fiduciary classification). It should not be read as a certification of compliance — DPDP's subordinate rules and enforcement guidance have continued to develop, and a current legal review is necessary before relying on this checklist for an actual compliance posture.
